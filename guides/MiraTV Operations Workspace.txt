MiraTV Operations Workspace (C:\MiraTV)
Purpose

This directory is the local operations execution workspace for MiraTV ingestion, synchronization, and catalog maintenance jobs.

It is designed to be:

Deterministic

Recoverable

Operator-friendly

Automation-safe

Future-observable (LLM / analytics-ready downstream)

This workspace executes jobs.
It does not serve content directly, and it does not perform analytics.

High-Level Architecture

The workspace is intentionally split into control, execution, state, data artifacts, and observability layers.

C:\MiraTV
├── bat/        → Legacy & compatibility launchers
├── jobs/       → Canonical job orchestration layer (WRAPPERS)
├── ps1/        → Execution / worker scripts (LOGIC)
├── ps1/lib/    → Shared infrastructure (OPS DB helpers)
├── export/     → Canonical output artifacts
├── json/       → Provider snapshots & normalized inputs
├── legacy_data/→ Cold fallback / rollback safety
├── tmp/        → Ephemeral working files
├── logs/       → Human-readable logs
├── state/      → Transitional local state (reserved)
├── run.bat     → Primary entrypoint

Entry Point
run.bat

Role: Primary execution entrypoint for automation and manual runs.

Called by:

Windows Task Scheduler

Operators

Higher-level orchestration

Delegates execution to job wrappers

run.bat should never contain business logic.

Job Orchestration Layer (/jobs)

This is the canonical execution layer.

All scheduled jobs must run through job wrappers in this directory.

Responsibilities

Acquire OPS DB locks

Record job state and semantic events

Enforce SAFE vs RISKY job policies

Invoke legacy worker scripts

Handle failures deterministically

Structure
jobs/
├── daily/
├── hourly/
├── weekly/
├── on_demand/
├── series/
├── legacy/   (deprecated / transitional)
└── *.job.ps1


Over time, all execution should converge on *.job.ps1 wrappers.

Execution / Worker Scripts (/ps1)

This layer does the actual work.

Scripts here:

Fetch

Transform

Import

Generate artifacts

Characteristics

Stateless where possible

No locking

No OPS DB access

File-based logging only

Exit codes must be meaningful

Examples

import_epg_chunked.ps1

import_live_channels.ps1

series_details_worker.ps1

Worker scripts assume they are called safely.

Shared Infrastructure (/ps1/lib)
ops_db.ps1

Critical file.

This module provides the only approved interface to the OPS database:

Job runs

Locks

Semantic events

Failures

Checkpoints

Rules

No business logic

No retries

No analytics

Short-lived DB connections only

If a script needs to persist execution state, it uses this module.

OPS State & Safety (/state)

Currently reserved for:

File-lock fallback (if ever required)

Emergency degraded-mode mechanisms

state/
└── locks/


Under normal operation, OPS DB locks are authoritative.

Data Artifacts
/export

Canonical outputs of jobs.

Examples:

EPG XML / JSON

Series catalogs

Live/VOD category exports

These files are:

Consumed by APIs

Synced to other systems

Used to generate decoy caches

/json

Normalized provider snapshots.

“Latest known good” inputs

Replayable

Diffable

Used by worker scripts

These are inputs, not truth.

/legacy_data

Cold fallback storage.

Known-good historical artifacts

Rollback safety

Never modified in-place

/tmp

Ephemeral workspace.

Chunk files

Intermediate transforms

Safe to delete at any time

Jobs must tolerate cleanup of this directory.

Logging (/logs)
Structure
logs/
├── jobs/        → Verbose execution logs
└── scheduler/  → Orchestration & scheduler logs

Logging Philosophy

File logs = human forensics

OPS DB = structured execution facts

Logs are disposable

DB records are authoritative

Legacy Launchers (/bat)

This directory contains older .bat entrypoints.

Status

Supported for now

Gradually deprecated

Should increasingly delegate to job wrappers

Database Boundaries (Critical)

This workspace interacts with one operational database only:

OPS DB: xpdgxfsp_ops

Purpose:

Locking

Job state

Semantic events

Failure recording

This workspace does not:

Perform analytics

Feed LLMs directly

Train models

Query META or INTEL databases

Those concerns are downstream.

Guiding Principles (Pinned)

OPS must be boring and safe

Execution must be deterministic

Locks are centralized

State transitions are explicit

Legacy scripts remain untouched

Intelligence never blocks operations

Operator Notes

If OPS DB is unavailable, jobs fail fast or skip safely

SAFE vs RISKY classification is enforced in wrappers

No job should ever run without a wrapper

No worker script should ever touch OPS DB directly

Future Extensions (Planned)

OPS → META export pipeline

Decoy cache generation artifacts

Read-only observability dashboards

LLM-assisted postmortems (downstream only)
