📺 MiraTV — Series Population & Normalization Pipeline (Authoritative)

Status: Active refactor
Scope: Series metadata, seasons, episodes (ingest → normalize → resolve)
Purpose: Stabilize database population for series content under unreliable provider payloads

1️⃣ Problem Statement (Why This Exists)

Series population is currently failing intermittently due to invalid / polluted provider payloads that cannot be parsed as strict JSON.

Observed issues include:

UTF-8 BOM contamination

CRLF + console artifact bleed

Embedded logging / Postman output inside payload

Partial JSON fragments

Valid JSON embedded inside invalid outer text

Conclusion:
Treating provider payloads as “strict JSON” is a false assumption.
Raw payloads must be treated as untrusted forensic text.

2️⃣ Core Design Decision (Locked)
❌ What We Do NOT Do

We do not reject payloads for being invalid JSON

We do not block ingestion on formatting

We do not parse episodes during series normalization

We do not rely on file names, MIME types, or extensions

✅ What We DO

Ingest everything into a raw table unchanged

Parse embedded JSON defensively

Normalize only series + seasons

Tag failures instead of aborting

Process one unit at a time

3️⃣ Canonical Series Ingest Flow
Provider API
   ↓
Raw Fetch (curl / Postman / Newman / Worker)
   ↓
Raw Storage (DB or filesystem)
   ↓
Raw Table (series_details_raw)
   ↓
Forensic Parser (text-based, not JSON-based)
   ↓
Series Normalization
   ↓
Seasons Normalization
   ↓
Episodes (handled later by separate worker)

4️⃣ Raw Storage Contract (Non-Negotiable)
📦 Table: series_details_raw

This table is append-only and never cleaned manually.

Required columns:

id

internal_series_id

provider_series_id

raw_provider_json (LONGTEXT)

parsed (0/1)

parsed_at

parse_error

Important:

raw_provider_json may contain garbage, console output, or partial payloads.
This is expected and allowed.

5️⃣ Parsing Philosophy (Forensic Mode)

The parser:

Treats raw payload as plain text

Searches for embedded JSON roots

Extracts the first valid balanced JSON object

Ignores everything else

Accepted JSON Roots

{"seasons": ... }

{"info": ... }

If no embedded payload is found:

Row is marked with parse_error

Processing continues with next row

Failure is data, not an exception.

6️⃣ Normalization Rules (Series Scope Only)
✅ Parsed & Stored

From info:

plot / overview

genre

rating

releaseDate

cover / cover_big

backdrop_path[]

From seasons:

season_number

name

episode_count

air_date

cover

❌ Explicitly Ignored

Episodes

Streams

Containers

Runtime playback URLs

These belong to separate workers.

7️⃣ Worker Behavior Contract

One row per execution

Transactional writes

Idempotent UPSERTs

No file movement

No deletion

No retries inside the worker

Status is written to DB, not logs

8️⃣ Known Failure Modes (Documented, Acceptable)
Failure	Result
Invalid outer JSON	Attempt forensic extraction
No embedded JSON	Mark parse_error = no_embedded_payload
Missing series ID	Mark parse_error = missing_series_id
Series not found	Mark parse_error = unknown_series
Partial seasons	Insert what exists

No failures block the pipeline.

9️⃣ Separation of Responsibilities (Critical)
Stage	Responsibility
Raw ingest	Capture everything
Parser	Extract structure
Normalizer	Populate relational tables
Episode worker	Create episodes
Resolver	Attach stream URLs
UI	Never blocks on backend lag
🔒 Final Principle (Locked)

Series population is a data-recovery problem, not a JSON problem.

The system must assume:

Providers are unreliable

Payloads are polluted

Correct data is embedded, not delivered cleanly

This document exists to prevent regressions to strict-JSON assumptions.
