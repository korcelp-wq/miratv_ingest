MiraTV Provider Normalization & Concurrent ID Mapping
Purpose

This document defines the provider normalization strategy for MiraTV and specifies the concurrent ID mapping model used to ingest, manage, and reason over Live, VOD, and Series content from multiple IPTV-style providers.

The goal is to:

Rapidly onboard structurally similar providers

Preserve provider-native identifiers

Establish MiraTV-owned canonical identities

Enable concurrent Live / VOD / Series alignment

Support downstream intelligence (recommendations, de-duplication, drift detection)

This is an application-level design, not a generic BYOS abstraction.

Design Principles

Providers Are Referential, Not Authoritative
Provider IDs describe claims, not truth. MiraTV owns canonical identity.

Normalize First, Reconcile Later
Content is ingested and normalized immediately; similarity and reconciliation are deferred.

Concurrent Mapping Is Explicit
Live, VOD, and Series IDs may coexist and overlap; mappings must support concurrency.

Fast Onboarding Beats Perfect Modeling
IPTV providers are similar enough that lookup-based normalization is preferable to deep abstraction.

Canonical IDs Are Stable
Provider churn must never invalidate MiraTV content IDs or embeddings.

MiraTV maintains normalized content tables. These are authoritative.

Content Types

live

vod

series

episode

Each content item receives a MiraTV canonical ID that never changes.

Provider Content Mapping Table (Core Artifact)
Table Name

provider_content_map

Purpose

Maps provider-native identifiers to MiraTV canonical content IDs while allowing:

Multiple providers → same content

One provider → multiple content types

Concurrent mappings (live + VOD + series)

Schema
provider_content_map
--------------------
id                      BIGINT PRIMARY KEY AUTO_INCREMENT
provider_name           VARCHAR(64) NOT NULL
provider_content_id     VARCHAR(128) NOT NULL
miratv_content_id       BIGINT NOT NULL
content_type            ENUM('live','vod','series','season','episode') NOT NULL
active                  BOOLEAN DEFAULT TRUE
created_at              TIMESTAMP DEFAULT CURRENT_TIMESTAMP
updated_at              TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP


UNIQUE(provider_name, provider_content_id, content_type)
INDEX(miratv_content_id)
Semantics

provider_content_id: Native ID from provider (string-safe)

miratv_content_id: Canonical MiraTV ID

content_type: Explicit — no inference

active: Allows graceful provider churn and soft retirement

Concurrent ID Support (Why This Table Exists)

Many providers expose:

A live channel

A VOD replay of the same channel

A series representing episodic captures

These may all be distinct IDs upstream.

Example
Provider	Provider ID	Type	MiraTV ID
A	101	live	9001
A	vod_55	vod	9002
A	series_9	series	9003

MiraTV does not collapse these automatically.

Instead:

Relationships are inferred later

Intelligence uses similarity, metadata, and usage

Ingest Rules
On Provider Onboarding

Inspect provider schema

Map provider fields to MiraTV normalized fields

Generate canonical content rows

Insert mapping rows into provider_content_map

On Provider Update

If provider ID unchanged → update metadata only

If provider ID changes → create new mapping row

Never overwrite canonical IDs

On Provider Removal

Set active = false

Do not delete canonical content immediately

Why This Is Better Than Global IDs

❌ Global ID enforcement

Slows onboarding

Breaks on provider quirks

Forces early reconciliation

✅ Mapping-based normalization

Fast onboarding

Provider-agnostic

Supports late binding

Interaction With Intelligence Layer
Vectorization

Vectors are generated from canonical content, not provider data

Provider changes do not invalidate embeddings

Cross-Provider Similarity

Same show across providers converges naturally in vector space

Mapping table preserves provenance

Duplicate / Alias Detection

Multiple canonical IDs can later be flagged as semantically identical

No destructive merge required

Anti-Patterns (Explicitly Avoided)

❌ Using provider IDs as primary keys

❌ Collapsing content at ingest time

❌ Letting vectors decide truth

❌ Over-abstracting providers behind heavy adapters

When to Reconcile Content

Reconciliation is optional and deferred.

Triggers may include:

High semantic similarity

Shared metadata (title, season, episode)

Editorial or human review

Reconciliation produces links, not deletions.

Summary (Canonical Rules)

Providers are referential

MiraTV owns identity

Mapping is explicit

Concurrency is allowed

Intelligence comes later

If two providers can be made equivalent by a lookup table, they are compatible enough to onboard.
