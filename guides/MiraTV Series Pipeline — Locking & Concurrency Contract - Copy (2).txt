📄 MiraTV Series Pipeline — Locking & Concurrency Contract

Status: AUTHORITATIVE
Scope: Series pipeline only
Audience: Humans, automation, future AI modules
Last Updated: (today)

1. Purpose

This document defines how locking works in the MiraTV Series pipeline to guarantee:

No duplicate processing

Safe restarts

Resume-after-crash behavior

Future-safe parallel processing

Deterministic automation

If code violates this document, the code is wrong.

2. Core Invariants (Non-Negotiable)

Two locks must NOT exist on the same series

At most one pipeline execution may run at a time

Locks protect execution, not data

Resets are explicit — never implicit

Database state is the single source of truth

3. Lock Types (Exactly Two)
3.1 Pipeline Lock (Global)

Purpose:
Prevents multiple pipeline runs from executing concurrently.

Scope:
Entire Series pipeline.

Implementation:

Single row in job_locks

Key: lock_name = 'series_pipeline'

Touched by:

Master runner only

Never touched by:

Workers

Series-level logic

3.2 Series Ownership (Per-Series)

Purpose:
Ensures only one worker may act on a series at any time.

Scope:
Single series.id.

Implementation:
No separate lock object.
The series row itself is the lock.

Ownership is encoded via atomic state transition:

details_state = 'IN_PROGRESS'

details_worker

details_locked_at

4. Series Ownership Lifecycle
4.1 Claim (Atomic)

A worker may proceed only if it successfully claims a series.

UPDATE series
SET
  details_state = 'IN_PROGRESS',
  details_worker = 'worker_name',
  details_locked_at = NOW(),
  details_last_attempt_at = NOW(),
  details_attempt_count = details_attempt_count + 1,
  details_error_code = NULL,
  details_error_msg = NULL
WHERE details_state IN (
  'NEED_FETCH',
  'RAW_CAPTURED',
  'PARSED',
  'NEEDS_NORMALIZE'
)
ORDER BY id
LIMIT 1;


Interpretation:

Rows affected = 1 → ownership granted

Rows affected = 0 → no eligible series or already owned

4.2 Release (Advance State)

Ownership is released only when state advances.

UPDATE series
SET
  details_state = 'NEXT_STATE',
  details_worker = NULL,
  details_locked_at = NULL
WHERE id = :series_id
  AND details_worker = 'worker_name';


The details_worker check prevents accidental unlocks.

4.3 Stale Ownership Recovery

If a worker crashes:

UPDATE series
SET
  details_state = 'NEED_FETCH',
  details_worker = NULL,
  details_locked_at = NULL
WHERE details_state = 'IN_PROGRESS'
  AND details_locked_at < NOW() - INTERVAL 30 MINUTE;


This guarantees self-healing without double execution.

5. What Is Explicitly Forbidden

❌ Local per-series lock files
❌ Multiple lock objects per series
❌ Workers proceeding without a successful claim
❌ Resetting state automatically
❌ Bypassing DB state due to “confidence”

6. Reset Semantics
6.1 Execution Reset (Safe)

Clears pipeline lock only.

Use when:

crash

reboot

hung terminal

Does not alter series data.

6.2 Pipeline Reset to Step 1 (Intentional)

Rewinds state flags to re-run logic.

Use when:

logic changes

provider improves data

controlled reprocessing is desired

Never deletes normalized data by default.

7. Parallel Processing Readiness

This lock model already supports parallelism.

Parallel execution is safe because:

Pipeline lock governs runs

Series ownership governs work units

Ownership is atomic

To enable parallel workers later:

Increase worker count

Do not change lock logic

8. AI Interaction Rules (Future-Facing)

Until explicitly promoted:

AI may read lock state

AI may report inconsistencies

AI may recommend resets

AI may not claim series

AI may not clear locks

AI may not modify state

This document is binding on AI behavior.

9. Enforcement Rule (Pinned)

If two workers ever process the same series concurrently,
the system is considered broken until proven otherwise.
