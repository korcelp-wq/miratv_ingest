## 2. Canonical Database Registry

All databases are centrally defined in a single registry file:

**Location**
/config/db.php


### Registry Design Principles

- No hard-coded DB names in workers or AI logic
- Logical DB roles resolved at runtime
- Supports relocation, sharding, cloning, or isolation without refactors
- Forms the structural backbone of CVI (Callosum Vector Integration)

## 3. Runtime Database Resolution

**Resolver File**
/config/db_resolve.php


### Usage Pattern

```php
$pdo = db_resolve('content');
$pdo = db_resolve('cortex');
$pdo = db_resolve('lake_vector');
Guarantees
Callers never reference credentials or DSNs directly

Runtime validation prevents silent misrouting

Unknown DB names throw hard exceptions

Ensures strict architectural boundaries between subsystems

This resolver is the only supported mechanism for DB access.

4. Backup Strategy (Authoritative)
Backup Scope
All registered databases must be backed up together to preserve system consistency.

Included:

Content DB (application runtime)

Cortex + Callosum (AI state and governance)

Knowledge + Vector lakes

Ops and inhibitor/governor matrices

Excluded:

information_schema (system DB)

Backup Script (PowerShell – Canon)
$MYSQL_BIN = "C:\Program Files\MySQL\MySQL Server 8.0\bin\mysqldump.exe"
$BACKUP_ROOT = "C:\miratv_backups"
$DATE = Get-Date -Format "yyyy-MM-dd_HH-mm-ss"

$DBS = @(
  "xpdgxfsp_content",
  "xpdgxfsp_cortex",
  "xpdgxfsp_callosum_matrix",
  "xpdgxfsp_lake_knowledge",
  "xpdgxfsp_lake_vector",
  "xpdgxfsp_ops",
  "xpdgxfsp_inhibitor_govenor_matrix",
  "xpdgxfsp_i_m_g_vector_context"
)

New-Item -ItemType Directory -Force -Path "$BACKUP_ROOT\$DATE" | Out-Null

foreach ($db in $DBS) {
  & $MYSQL_BIN `
    -u xpdgxfsp `
    -p `
    --single-transaction `
    --routines `
    --events `
    --databases $db `
    > "$BACKUP_ROOT\$DATE\$db.sql"
}

Write-Host "✔ MiraTV DB Backup Complete: $DATE"
Backup Characteristics

Transaction-safe

Schema + data

Stored routines included

One folder per canon snapshot
