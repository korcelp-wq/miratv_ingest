2. Current Problem

Episodes do not display in the app.

The logs show:

JsonSyntaxException:
Expected BEGIN_OBJECT but was BEGIN_ARRAY
path $.episodes.[0].info

Meaning:

The provider API returns inconsistent JSON structures.

Example variations seen in providers:

Case A
"info": { ... }
Case B
"info": [ ... ]
Case C
"info": null
Case D
no info field at all

The Android client uses Gson models expecting a strict object, so when the provider returns an array, parsing fails.

3. Secondary Issue

The local database currently contains very few episodes.

Reason:

50,000+ series exist
but ingestion/grinder hasn't processed them yet

Therefore the client frequently hits the provider fallback path, which currently fails due to the JSON issue.

4. Key Design Decision (User Direction)

User requested:

Fix CLIENT first
Fix DATABASE second

Meaning:

1️⃣ The client must play episodes even if the DB is empty
2️⃣ The ingestion system can be repaired later

5. Immediate Fix Strategy

The current Retrofit/Gson pipeline is too strict.

Instead:

Provider fallback must return raw JSON

Then parse episodes defensively.

Parsing rules

Episodes may appear as:

episodes: []
episodes: {}
episodes: { season_number : [] }
episodes: { season_number : {} }
episodes: single object

All must normalize to:

List<Episode>
6. Episode Normalization Rules
If episodes is missing
episodes = []
If episodes is array
episodes = response.episodes
If episodes is object keyed by season

Example:

episodes : {
   "1": [ ... ],
   "2": [ ... ]
}

Flatten to list.

If single episode object

Wrap:

episodes = [episode]
7. Important Edge Case

Providers sometimes return a single episode only.

Example:

episodes: {
  "1": {
     "id": 123
  }
}

Must normalize to:

episodes = [ episode ]
