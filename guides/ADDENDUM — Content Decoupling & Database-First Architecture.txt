📘 ADDENDUM — Content Decoupling & Database-First Architecture

Status: LOCKED / AUTHORITATIVE
Applies To: Movies, Series, Live TV, EPG
Effective: Immediately
Supersedes: Any direct UI dependency on provider list endpoints

1️⃣ Architectural Mandate (Non-Negotiable)

The MiraTV application MUST NOT block, wait, or depend on provider APIs for UI rendering.

All user-facing content lists (Movies, Series, Live Channels, EPG) are served exclusively from MiraTV-controlled infrastructure (database + internal API).

Provider systems (Xtream / XC / equivalent) are treated as:

Asynchronous upstream data sources

Non-deterministic

Latent

Potentially unavailable

They are never used as a real-time UI dependency.

2️⃣ Canonical Data Flow (Locked)
Provider (Xtream / XC)
   ↓  (async ingest, polling, or diff)
Ingest Layer (cron / workers)
   ↓
MiraTV Database (normalized, indexed)
   ↓
MiraTV Internal API (fast, paginated)
   ↓
Android App (instant UI)


The app never calls provider list endpoints directly.

3️⃣ Scope of Database Mirroring

The following content domains are fully mirrored into the MiraTV database:

🎬 Movies

Categories

Metadata

Artwork URLs

Codec/container info

IDs only (no stream URLs)

📺 Series

Series metadata

Seasons

Episodes

Episode IDs and container extensions

🔴 Live TV

Channel lists

Categories

Stream IDs

Logos

🗓️ EPG

Parsed XMLTV data

Channel-indexed programs

Time-windowed slices

4️⃣ Explicit Playback Boundary (Critical)

Playback URLs are NEVER stored in the database.

The database stores only:

IDs

Metadata

Container extensions

Playback URLs are constructed at play time only, using the provider contract:

Live:

/live/{username}/{password}/{stream_id}.ts


Movies:

/movie/{username}/{password}/{movie_id}.{ext}


Series Episodes:

/series/{username}/{password}/{episode_id}.{ext}


Provider-side redirects, tokens, and signing are:

Allowed

Expected

Handled by the player layer

5️⃣ Incremental Update Model (No Blocking)

Content ingestion follows an incremental, non-blocking model:

last_modified fields are authoritative

Only changed records are refreshed

Full reloads are avoided unless forced

Ingest jobs run independently of user activity

At no point does the application wait for:

Provider availability

Ingest completion

Background refresh cycles

6️⃣ App Behavior Guarantees (User-Facing)

As a direct result of this architecture, the app guarantees:

No blank screens due to provider latency

No UI freezes from large payloads

No startup waits

No “loading everything” states

Deterministic navigation timing

All UI queries are:

Paginated

Indexed

Served in milliseconds

7️⃣ Infrastructure Compatibility (Confirmed)

This architecture is explicitly designed to operate on:

cPanel hosting

MySQL / MariaDB

PHP-based APIs

Cron-driven ingest jobs

Streaming traffic is never handled by MiraTV infrastructure.

This ensures:

High stability

Low resource usage

Linear scalability

No premature cloud dependency

8️⃣ Strategic Classification (Locked)

With this addendum in force, MiraTV is formally classified as:

A middleware-driven IPTV platform with a decoupled, deterministic client — not a thin provider client.

This distinction is intentional and foundational.

9️⃣ Invariant (Do Not Violate)

The app may observe provider state indirectly,
but the database is the sole authority for content discovery.

Any implementation that violates this principle is considered architecturally invalid.
